\section{Dashboard User Guide}

This appendix is a short operating guide for a reader who wants to launch the dashboard and reproduce the inspection workflow evaluated in \S6, without reading the implementation chapter first.

\subsection{Requirements and Launch}

The dashboard requires Python 3.10+ and the dependencies in \path{requirements.txt}. From the repository root:

\begin{verbatim}
python -m venv .venv && source .venv/bin/activate
python -m pip install -r requirements.txt
python src/data/build_canonical.py
python src/data/build_borough_summary.py
flask --app src.api.app:create_app run --debug --port 5057
\end{verbatim}

The two build scripts regenerate the untracked parquet artefacts from the raw LFB workbook, which must be present under \path{data/} (it is excluded from version control for licensing reasons; its source URL and SHA-256 are recorded in the provenance sidecar). The boundary GeoJSON is tracked directly. The dashboard is then served at \path{http://127.0.0.1:5057/dashboard}.

\subsection{Controls and Linked Behaviour}

The interface holds one shared selection state: \emph{borough $\times$ month $\times$ incident group}. It can be changed three ways: the three select controls at the top, clicking a borough on the map, or clicking a row in the ranking table. Any change re-renders all four panels together:

\begin{itemize}
    \item \textbf{Borough map (choropleth).} Fill colour encodes the six-minute exceedance share for the selected month and incident group. Hovering a borough shows its incident count, median and p90 response times, denominator-qualified exceedance share, and precise-coordinate coverage.
    \item \textbf{Monthly trend panel.} Incident count and exceedance share for the selected borough and group across the full 26-month slice.
    \item \textbf{Response-distribution panel.} Median-to-p95 response-time ranges for the three incident groups in the selected cell, against the dashed six-minute reference line.
    \item \textbf{Ranking table.} Boroughs ordered by exceedance share for the selected month and group, with exact values and the per-borough precise-coordinate coverage column.
\end{itemize}

If the selected combination has no data, an amber empty-state banner replaces silent blank panels; adjusting any filter recovers a populated view.

\subsection{How to Read the Values}

Exceedance shares are computed over incidents \emph{with a recorded first-pump attendance time}, not over all incidents; the ranking table's coordinate-coverage column reports how much of each borough's data carries precise coordinates (37.9\% overall). A defensible comparison therefore reads three things together: the exceedance value, the incident count behind it, and the coverage share. A high-exceedance borough with a small count or low coverage deserves a weaker claim.

\subsection{What the Station Evidence Does and Does Not Support}

Station-related evidence in this project is report-side and API-side, not an interactive map layer: the current dashboard screen does not render station markers, and the footprint-scenario UI toggle is not implemented in this iteration (\S6.3). The assignment-footprint artefact locates each station's \emph{historical incident responsibility centroid}, not its building; the public station-address table adds postcode-level proximity context only. Neither supports coverage, routing, or relocation conclusions (\S5.5--\S5.6, \S6.2).

\subsection{Example Task}

To repeat the comparison task in Table~\ref{tab:task-evaluation-matrix}: select \emph{Fire} as the incident group, click \emph{Hillingdon} on the map, note its exceedance share and trend, then click an inner-London row such as \emph{Tower Hamlets} in the ranking table and compare the two boroughs' exceedance values, incident counts, and coordinate-coverage shares before deciding which claims the data supports.

\subsection{Known Limitations}

The current dashboard still has no one-click reset to the all-London default, and the fuller uncertainty encodings described in \S6.5 remain second-iteration work. The station-related evidence is also intentionally separated from the live dashboard screen because the project does not implement a routing-grade station-coverage model.
